# Bridge — AWS Ubuntu deployment

Production-grade single-VM deployment of Bridge to AWS EC2 (Ubuntu 22.04 or
24.04 LTS).  Backend runs behind Nginx + Let's Encrypt HTTPS.  **Data is
stored in Supabase (Postgres)** — there is no database to install on the VM.

## Prerequisites

1. **EC2 instance** — Ubuntu 22.04 LTS or 24.04 LTS. Minimum recommended:
   - **t3.medium** (2 vCPU, 4 GiB RAM) — dev / < 20 concurrent users
   - **t3.large** (2 vCPU, 8 GiB RAM) — production small teams
   - **c6i.xlarge** (4 vCPU, 8 GiB RAM) — production heavy
2. **Elastic IP** attached to the instance (so the DNS record never breaks).
3. **DNS A record** for your domain (e.g. `bridge.mycompany.com`) → the EIP.
4. **Security group** with these inbound rules:
   - SSH (22) from your IP
   - HTTP (80) from 0.0.0.0/0
   - HTTPS (443) from 0.0.0.0/0
5. **Supabase project** — create one at https://supabase.com. You will need:
   - Project URL, e.g. `https://abcdefgh.supabase.co`
   - **service_role** secret key (Project Settings → API → service_role)
   - (optional) direct Postgres connection string for the deploy script to
     apply the schema automatically (Project Settings → Database → Connection
     string → Transaction pooler). If you skip this, just paste
     `backend/schema.sql` into the Supabase SQL editor once.
6. LiveKit + OpenAI + Stripe accounts (LiveKit Cloud works out of the box; if
   you self-host LiveKit you'll additionally need to expose the LiveKit
   media ports — see LiveKit docs).

## First-time deploy

From your local machine:

```bash
# 1. Zip and copy the repo (or use git)
rsync -avz --exclude node_modules --exclude .git --exclude _legacy \
  --exclude frontend/build --exclude '**/__pycache__' --exclude '**/.venv' \
  /path/to/app/  ubuntu@YOUR_EIP:/tmp/bridge/

# 2. SSH in and run the installer
ssh ubuntu@YOUR_EIP
sudo mv /tmp/bridge /opt/bridge-src
cd /opt/bridge-src
sudo bash deploy/setup.sh bridge.mycompany.com ops@mycompany.com
```

The installer will:

- install Python 3 (+ dev headers), Node 20, Nginx, certbot, and **uv** (fast
  Python package manager — no `python3-venv`/`python3-pip` needed)
- create the `bridge` service user and `/opt/bridge`
- ensure a **wheel-compatible Python** is used for the venv (3.10 / 3.11 /
  3.12 / 3.13 / 3.14). `asyncpg==0.31.0` ships cp39–cp314 wheels, so Python
  3.14 is fully supported. setup.sh prefers the **newest available system**
  interpreter (created with `--python-preference only-system` so uv never
  substitutes its managed `python-build-standalone`, which is missing
  `_sysconfigdata_` on some Ubuntu/glibc combos). If the host has no
  suitable system Python (e.g. Ubuntu 'resolute' / newer AMIs where the
  deadsnakes PPA is also unsupported), setup.sh falls back to
  **uv-managed Python 3.14** (`uv python install 3.14`). Backend deps —
  including `emergentintegrations` — are installed into the venv.
- write `/opt/bridge/backend/.env` with a **Supabase** template
- attempt to apply `backend/schema.sql` to Supabase (best-effort; prints the
  SQL for you to paste into the Supabase SQL editor if `SUPABASE_DB_URL`
  isn't set)
- build the React frontend (`yarn build`) into `/opt/bridge/frontend/build`
- install a `bridge-backend.service` systemd unit (**no mongod dependency**)
- configure Nginx to serve the static build + reverse-proxy `/api/`
- get an HTTPS certificate from Let's Encrypt
- start the backend and run a `/api/health` smoke test

Total runtime is ~5–8 minutes.

> ⚠️ If the installer detects placeholder secrets in `.env`
> (`SUPABASE_URL`, `SUPABASE_SECRET_KEY`, `LIVEKIT_*`) it will **refuse to
> start the backend** and print a clear message. This prevents the old
> symptom where uvicorn would crash-loop silently because `supa.py` raises
> `KeyError` on a missing `SUPABASE_URL`.

## After first install — set your real API keys

```bash
sudo nano /opt/bridge/backend/.env
```

Fill in real values for:

- `SUPABASE_URL`, `SUPABASE_SECRET_KEY` (required — backend won't start
  without them)
- `SUPABASE_DB_URL` (optional — only needed if you want the deploy script to
  apply the schema for you; otherwise paste `backend/schema.sql` into the
  Supabase SQL editor once)
- `LIVEKIT_URL`, `LIVEKIT_API_KEY`, `LIVEKIT_API_SECRET`
- `OPENAI_API_KEY`
- `STRIPE_API_KEY` (use `sk_live_…` for prod)
- (optional) change `FIXED_ADMIN_EMAIL` / `FIXED_ADMIN_PASSWORD` — the initial
  password is auto-generated by the installer; write it down BEFORE closing
  the shell.

Then reload:

```bash
sudo systemctl restart bridge-backend
```

## Apply the Supabase schema

Two options:

**Option A — SQL editor (simplest, no extra setup):**
1. Open your Supabase project → SQL Editor → New query.
2. Paste the contents of `/opt/bridge/backend/schema.sql`.
3. Run. It is idempotent (`create table if not exists`), so re-running is
   safe.

**Option B — programmatic (set `SUPABASE_DB_URL` in `.env`):**
```bash
sudo -u bridge bash -c "cd /opt/bridge/backend && \
  .venv/bin/python scripts/init_supabase.py"
```
This connects via asyncpg and applies `schema.sql` directly. The connection
string is the **Transaction pooler** URL from Supabase Dashboard → Project
Settings → Database, e.g.
`postgresql://postgres.<ref>:<password>@aws-0-<region>.pooler.supabase.com:6543/postgres`.

## Troubleshooting — Python / venv errors

These are the two uv/venv errors you may hit during `setup.sh`, and what the
installer now does to prevent them:

### `error: Python installation is missing a '_sysconfigdata_' file`

**Cause:** uv's managed `python-build-standalone` is missing `_sysconfigdata_`
on some Ubuntu/glibc combinations, so `uv venv` fails outright.

**Fix (already in setup.sh):** the venv is always created with
`uv venv --python <system-python> --python-preference only-system`, which tells
uv to use the **system** interpreter and never substitute its managed build.

### `Failed to build asyncpg==0.30.0` / `Python.h: No such file or directory`

**Cause:** the venv was created with **Python 3.13 or 3.14**, and
`asyncpg==0.30.0` has **no prebuilt wheel** for those (it only ships
cp38–cp313 wheels). uv then tries a source build, which fails because
(a) `Python.h` dev headers aren't installed and (b) the Cython-generated C
in 0.30.0 isn't 3.14-compatible.

**Fix (already in setup.sh + requirements.txt):**
1. `requirements.txt` was bumped from `asyncpg==0.30.0` to
   **`asyncpg==0.31.0`**, which explicitly adds Python 3.14 support
   (cp314 wheels). No source build is needed on any Python 3.9–3.14.
2. setup.sh prefers the **newest available system** Python in
   **{3.14, 3.13, 3.12, 3.11, 3.10}**, created with
   `--python-preference only-system` so uv never substitutes its own
   managed build (avoids the `_sysconfigdata_` bug seen on uv's 3.11 build).
3. If no suitable system Python is installed (e.g. Ubuntu 'resolute' or a
   newer AMI where the deadsnakes PPA is also unsupported), setup.sh falls
   back to **uv-managed Python 3.14** (`uv python install 3.14`). asyncpg
   0.31.0 ships cp314 wheels, and uv's 3.14 managed build is stable
   (the `_sysconfigdata_` bug was 3.11-specific).
4. `python3-dev` is installed as a safety net for any other package that
   might need to fall back to a source build.

### Manual recovery from a broken venv

If you hit either error on an older copy of `setup.sh` and have a stale
`.venv` left behind, the cleanest recovery is:

```bash
sudo rm -rf /opt/bridge/backend/.venv
sudo bash /opt/bridge/deploy/setup.sh <domain> <email>
```

Or, to rebuild just the venv without re-running the whole installer (this
works on any host, including ones with no system Python 3.10–3.14 —
uv downloads its own managed 3.14):

```bash
sudo -u bridge /usr/local/bin/uv python install 3.14
sudo -u bridge bash -c '
  cd /opt/bridge/backend
  rm -rf .venv
  export UV_CACHE_DIR=/opt/bridge/.cache/uv
  /usr/local/bin/uv venv --python 3.14 .venv
  /usr/local/bin/uv pip install --python .venv/bin/python -r requirements.txt \
    --extra-index-url https://d33sy5i8bnduwe.cloudfront.net/simple/
'
sudo systemctl restart bridge-backend
```

`redeploy.sh` also detects a stale venv (Python outside 3.10–3.14, or
missing) and recreates it from the newest available Python automatically,
so `sudo bash /opt/bridge/deploy/redeploy.sh` is enough after a `git pull`.

## Ongoing operations

| Task                     | Command                                            |
|--------------------------|----------------------------------------------------|
| Redeploy after code push | `sudo bash /opt/bridge/deploy/redeploy.sh`         |
| Restart backend          | `sudo systemctl restart bridge-backend`            |
| Backend logs (live)      | `sudo journalctl -u bridge-backend -f`             |
| Backend logs (last 200)  | `sudo journalctl -u bridge-backend -n 200 --no-pager` |
| Add a Python package     | `sudo -u bridge bash -c "cd /opt/bridge/backend && \`<br>`/usr/local/bin/uv pip install --python .venv/bin/python <pkg>"` |
| Nginx logs               | `sudo tail -f /var/log/nginx/{access,error}.log`   |
| Restart nginx            | `sudo systemctl reload nginx`                      |
| Renew SSL manually       | `sudo certbot renew --nginx`                       |
| Disk usage               | `df -h /`                                          |

`redeploy.sh` now ends with a `/api/health` smoke test and will print the
last 40 journal lines if the backend fails to come up — so a bad `.env`
shows up immediately instead of a silent crash-loop.

## Upgrading from the old MongoDB-based deploy

If you previously deployed Bridge with MongoDB:

1. **Migrate your data** from Mongo to Supabase first (export each Mongo
   collection → insert into the matching Supabase table per
   `backend/schema.sql`). The backend code already writes only to Supabase,
   so there is no dual-write path.
2. Stop the backend: `sudo systemctl stop bridge-backend`.
3. Pull the new code and re-run `sudo bash /opt/bridge/deploy/setup.sh
   <domain> <email>`. The installer will:
   - append the Supabase vars to your existing `.env` (your old LiveKit /
     OpenAI / Stripe / admin values are preserved),
   - replace the systemd unit (removing the `Requires=mongod.service`
     dependency that previously prevented startup on MongoDB-less hosts),
   - apply the schema if `SUPABASE_DB_URL` is set.
4. You can leave `mongod` running or stop it — Bridge no longer touches it:
   `sudo systemctl disable --now mongod` (optional).

## Scaling notes

- The FastAPI service runs with `--workers 2` by default. For a 4-vCPU box
  bump this to `--workers 4` in `/etc/systemd/system/bridge-backend.service`.
- All the heavy media processing (WebRTC / SFU / translation LLM / TTS) is
  handled by **LiveKit Cloud**, **OpenAI**, and **Supabase**, not by this
  VM — so the VM itself has very low CPU/bandwidth needs.  A single
  t3.medium can serve hundreds of thousands of API requests per day.
- Supabase autoscales its Postgres pooler; if you outgrow the free tier,
  upgrade the Supabase project — no code changes needed.
- Static React build is served directly by Nginx (cached for 1 year on
  hashed chunks), so add a CloudFront distribution in front of Nginx if
  you need a global CDN.

## Firewall

The installer enables UFW with only OpenSSH + HTTP + HTTPS allowed. AWS
Security Groups should mirror this; you don't need to open any other ports
because LiveKit media traffic goes browser ↔ LiveKit Cloud directly, not
through your VM.

## Rollback

The installer keeps the previous `/opt/bridge` state via rsync's `--delete`
behaviour. To roll back, just re-rsync a previous snapshot of `/opt/bridge`
and run `redeploy.sh`. For a full rollback of certs / nginx config, restore
from your regular EC2 snapshot.

## Uninstall

```bash
sudo systemctl disable --now bridge-backend
sudo rm /etc/systemd/system/bridge-backend.service
sudo rm /etc/nginx/sites-enabled/bridge /etc/nginx/sites-available/bridge
sudo systemctl reload nginx
sudo rm -rf /opt/bridge
sudo userdel bridge
# Optionally drop the Supabase project / schema from the Supabase dashboard.
```
